Safeguards & Enterprise Trust

Customers are not developers. The system must protect them like they aren't.

We are asking IT Managers to trigger code generation and deployment. Every safeguard below exists to make that safe — technically and legally.

⚠️
Sandbox Before Production — Non-Negotiable
Every integration runs in an isolated sandbox environment before it is allowed anywhere near production data. The Deploy button is physically locked until all sandbox stages pass. No override exists for first-time integrations. This is not a setting — it is architecture.
🔒
Input Safety

Scope Minimiser

AI requests only the permissions required for the stated use case. Over-permissioned scopes are flagged and rejected before code is shown to the user.

🛡️
Generation Safety

SAST + Security Pass

Static analysis runs automatically on every generated file. Code with hardcoded secrets is regenerated, never shown. Code Llama runs a dedicated security audit pass.

📦
Environment Safety

Zero-Trust Sandbox

Containerised execution with outbound network blocked except to CloudEagle's mock server. Three stages must all pass before Approve is enabled.

👤
Human Approval Gate

Security Lead Sign-off

Each flagged item requires individual checkbox acknowledgment — not one "I agree" click. Approvals under 60 seconds on flagged integrations trigger a mandatory re-review prompt.

🔐
Data Privacy

PII Masking + AES-256

All sensitive data is scrubbed locally before any LLM call. API credentials use AES-256 encryption and are stored in a secrets vault — never passed to AI APIs.

📋
Audit & Compliance

Complete Signed Trail

Every action — generation, review, approval, deployment — is logged with timestamp and actor. No-training clause enforced via enterprise API agreements with all LLM providers.

AI Model Strategy

Three models. Each chosen for a specific job.

A single model can't do all of this well. The architecture separates reading, writing, and auditing — and assigns the best model to each role.

Step 1–2
Architect
Claude Sonnet 4
Doc Reading & Planning

Why: Best-in-class 200K token context window — essential for reading multi-page API documentation in full. Reliable instruction-following and structured JSON output. Does not hallucinate endpoints because it reasons from what it actually reads.

→ Reads and parses API documentation URL
→ Extracts auth flows, endpoints, schemas
→ Outputs structured integration plan in plain English
→ Human reviews plan before any code is written
Step 3–4
Developer
GPT-4o
Code Generation

Why: Market-leading for idiomatic, production-grade code. Familiar with existing API client patterns across languages. Generates complete auth flows, pagination handlers, exponential backoff, and error handling — not skeleton code.

→ Generates API client from approved plan
→ Writes auth setup, pagination, error handling
→ Adds logging and rate-limit resilience
→ Hands off to Code Llama for security audit
Step 5
Auditor
Code Llama (fine-tuned)
Security Validation

Why: Specialised for code analysis — not general reasoning. Detects anti-patterns, over-permissioned OAuth scopes, hardcoded secrets, and missing retry logic. Runs on cheaper compute than GPT-4o, fine-tuned to CloudEagle's internal security standards.

→ Scans every generated file before sandbox
→ Flags over-permissioned scopes
→ Rejects code with hardcoded credentials
→ Passes clean code to sandbox environment
Why not one model for everything? Separation of concerns. An architect who also writes and audits their own code misses their own mistakes. Three specialised models with human checkpoints between each stage is more reliable, more auditable, and cheaper to run than one general model doing it all.
