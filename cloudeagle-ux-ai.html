A
Three input modes — pick the one you have
Not every IT Head has a URL. Some will have an OpenAPI spec from their vendor. Some will have a Postman collection. All three paths lead to the same outcome. The default is URL because it requires the least technical knowledge.
Reduces friction
B
The reassurance notice — set the expectation early
The blue notice tells Rohan that nothing is built until he sees and confirms what the AI understood. This is the trust-building moment before any technical complexity appears. It answers the question "what will happen next?" before he has to ask.
Builds trust
C
Plain-language use case selection
No API endpoints visible here. No technical configuration. Rohan picks what he wants to accomplish in plain English. The system translates his choice into the specific API calls needed — that translation is invisible to him at this stage.
Non-technical entry
D
Primary CTA — single, unambiguous
One button. No secondary options competing for attention. The design is intentionally sparse — the only decision on this screen is "do I want to proceed?" Everything else comes after.
User goal on this screen
Tell the system which tool to connect and what to do with it
Primary CTA
Analyse Documentation
Key decision
Which tool? Which use case? Which input method?
Risk reduced
Confusion about what to do first. Solved by single, clear CTA and plain-language options.

A
Plan before code — the cardinal rule of this product
The AI summarises its understanding in a human-readable format before a single line of code is generated. If the AI misunderstood the documentation (e.g. confused v1 and v2 endpoints), Rohan can catch it here — not after deployment.
Safety gate #1
B
Plain language, not API jargon
"Cursor-based pagination" becomes "pages of results, navigated using a token." Auth types are shown with a plain description, not an RFC number. Rohan doesn't need to know what OAuth 2.1 is — he needs to know the system found a secure login method.
Accessible to non-technical users
C
Two CTAs — proceed or flag a problem
The secondary CTA "Something looks wrong" is deliberately available. It opens a feedback field where Rohan (or his engineer colleague) can describe the inaccuracy. This feedback loop improves the AI's doc parsing over time and prevents silent errors from propagating.
D
Data objects shown — not endpoint paths
The screen shows what kind of data exists in the tool (Users, Events, EventTypes) — not the raw API paths. This is intentional: Rohan thinks in terms of data and business objects, not URL structures. The technical mapping is handled in the next screen.
User goal
Verify the AI understood the tool correctly before proceeding
Primary CTA
Confirm & Continue
Key decision
Is this understanding accurate? Yes → proceed. No → flag for correction.
Risk reduced
AI misunderstanding the API structure and generating incorrect code. Caught here before code is written.

A
Numbered steps in plain English — not pseudocode
Each step describes the integration logic the way Rohan would describe it to a colleague — not the way an engineer would write it in comments. "Collect all meetings page by page until complete" is a human description of pagination. This is intentional.
Human-readable logic
B
Permission summary — visible before approval
Before Rohan approves, he sees exactly what permissions the integration will request. If it says "admin access" when only read access was needed, he can flag it. This is where the scope minimiser's output surfaces to the user.
Scope transparency
C
Security stated plainly — not assumed
"Credentials stored securely — never in the code" appears as step 05. This is not a disclaimer in fine print — it is a named step in the plan. The security decision is visible and explicit, not hidden.
D
This is safety gate #2
If the plan looks wrong — wrong permissions, wrong data collected, wrong logic — Rohan can go back and adjust the use case selection. Catching an error here costs 2 minutes. Catching it after deployment costs hours and trust.
Safety gate #2
User goal
Confirm the integration logic is correct before code is written
Primary CTA
Approve Plan — Generate Code
Key decision
Does this plan do what I need? Are the permissions right?
Risk reduced
Code generated for the wrong use case or with excessive permissions. Both caught here.

Generated Code — with explanations, not just syntax

The code is shown with inline plain-English comments explaining why each decision was made. Rohan doesn't need to read the code — but his engineering team can review it and understand the AI's reasoning without documentation.

# CloudEagle AI Integration Builder
# Tool: Calendly v2 · Scope: users:read, events:read
# Security: credentials via vault, never in code

import os, time, requests

# Token fetched from CloudEagle vault at runtime
TOKEN = os.environ["CALENDLY_TOKEN"]

# Sandbox/production switch — one variable controls it
BASE = ("https://mock.cloudeagle.ai/calendly"
        if os.getenv("CE_ENV") == "sandbox"
        else "https://api.calendly.com")

# Handles rate limits — waits and retries automatically
def call(endpoint, **params):
  for attempt in range(3):
    r = requests.get(f"{BASE}{endpoint}", ...)
    if r.status_code == 429:
      time.sleep(2 ** attempt)
    elif r.ok: return r.json()
Test in Sandbox →
A
Every non-obvious line has a comment
Comments are written in plain English — not developer shorthand. "Handles rate limits — waits and retries automatically" is more valuable to an IT Head's engineering team than "exponential backoff on 429." The code is self-documenting for the review step.
Explainable AI
B
Security badges — not buried in the code
The green tags at the top ("Security ✓", "Scope ✓") show that the automated checks passed. These are not cosmetic — they represent the output of the SAST scanner and scope minimiser. An IT Head should be able to see "this passed security review" without reading the code.
Trust signal
C
The sandbox/production switch is visible
Rohan can see the one variable that controls whether the code runs against real data or test data. This transparency is deliberate — it shows the system is designed for safe testing, not blind deployment.
D
Rohan's action on this screen: nothing
He doesn't need to read or approve the code itself. His action is simply to proceed to sandbox testing. The code review (if needed) happens with his engineering team — the download option supports that handoff without slowing the primary flow.
User goal
Have confidence the code is safe and correct — without reading it
Primary CTA
Test in Sandbox
Key decision
Does this look right to me? Should I involve my engineering team before testing?
Risk reduced
Opaque AI output that users can't assess. Solved by security badges, plain-English comments, and download for team review.

A
Split pane — code running + data preview simultaneously
Left pane: the terminal showing each test step as it runs, in real time. Right pane: a preview of what data would be captured in production. Rohan sees both — the process and the output — without switching screens.
Full transparency
B
The rate limit simulation is visible
The amber line "429 simulated → retry" shows the error handling working correctly. Rohan can see that when Calendly says "slow down," the system automatically waits and tries again. This is a feature being demonstrated, not hidden.
C
Mock data is clearly labelled as fake
"This data is fake" appears directly in the data preview. This is not assumed — it is stated. The fake names use @mock.ai email domains. There is zero chance of Rohan confusing mock results with real organisational data.
No data confusion
D
Safety gate #3 — no bypassing sandbox
The "Proceed to Approval" button only appears after all core tests have run. The one amber warning (field mapping review) is surfaced clearly — Rohan cannot miss it and cannot skip it. This is not a warning banner — it is a step in the flow.
Safety gate #3 — mandatory
User goal
Confirm the integration works correctly before touching real data
Primary CTA
Review & Proceed to Approval
Key decision
Did all tests pass? Is the data preview capturing what I expected?
Risk reduced
Unvalidated code reaching production. Sandbox is mandatory and non-bypassable for first-time integrations.

A
Checklist — not a single sign button
Each item in the checklist must be individually checked. The approver cannot sign off by clicking one "I agree" button. For flagged items (like the amber KYC field), the checkbox cannot be ticked until the user has addressed the flag. This prevents rubber-stamping.
Prevents blind approval
B
Scope audit — what was and wasn't requested
Not just what permissions are being requested — also showing what was NOT requested. "admin:write — Not requested ✓" is as important as showing the read permissions. It proves the system is following minimum privilege, not just claiming it.
C
Named approver creates accountability
The approver's name and role appear in the audit log. This is not just about compliance — it is about accountability within the organisation. If an integration is questioned later, there is a clear record of who reviewed and approved it, and when.
Audit trail
D
The green Deploy button only appears when checklist is complete
As long as any checklist item is unchecked, the Deploy button is disabled — not greyed out, not hidden. Disabled and visually clear. The user knows exactly why they cannot proceed and exactly what they need to do.
User goal
Formally review and approve the integration before it goes live
Primary CTA
Sign & Deploy to Production
Key decision
Am I confident enough to approve this? Have I reviewed everything that was flagged?
Risk reduced
Unapproved or unreviewed integrations reaching production. Deploy is locked until every checklist item is confirmed individually.

A
Three numbers that matter to Rohan's CFO
Users governed, setup time, engineering tickets. These are the numbers Rohan will use to justify the platform investment. The success screen frames the outcome in his language — not the system's language.
Business value framing
B
"What happens next" — Rohan doesn't have to do anything
The four bullet points tell Rohan that the system takes care of everything from here. He doesn't need to schedule a sync, set up monitoring, or check back in. This is the "set and forget" promise — backed by EagleEye's continuous monitoring.
C
API change notification — mentioned on the success screen
"If Calendly's API changes, you'll be notified before anything breaks." This is the versioning promise stated in plain English at the moment of completion. It sets the expectation that the integration is not a one-time build — it is a managed, monitored connection.
Versioning promise
D
Two CTAs — next logical actions
The primary CTA takes Rohan to where the value is delivered: the governance dashboard. The secondary CTA opens the door to expanding coverage. Both are natural next steps — the screen doesn't leave him wondering what to do after a successful integration.
User goal
Confirm success and understand what the system will do from here
Primary CTA
View Governance Dashboard
Key decision
What do I do next? Connect another tool? Review the data?
Risk reduced
Post-deployment confusion or neglect. Solved by stating what the system monitors automatically and surfacing the value outcome immediately.
